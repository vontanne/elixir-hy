## Ֆունկցիաների լրացուցիչ հնարավորությունները

Նախորդ երկու հոդվածում ծանոթացանք ֆունկցիաների երկու տարբեր ձևերին՝ [մոդուլային](./modules_and_functions.md) և [անանուն](./anonymous_functions.md)։ Այսօր կծանոթանանք `&` օպերատորին, ինչպես նաև կզարգացնենք մեկ այլ թեմա՝ pattern matching-ը հենց ֆունկցիայի արգումենտներում։

### «Բռնել» գոյություն ունեցող ֆունկցիան

Մինչ այժմ `անուն/arity` գրառումը մեզ պետք էր գալիս միայն ֆունկցիայի մասին խոսելիս՝ `String.length/1`, `Map.get/3`։ `&` օպերատորի շնորհիվ այդ նույն գրառումը դառնում է լիարժեք արտահայտություն. դնում ես `&` նշանը ֆունկցիայի անվան առջև, եւ ստանում ես հենց այդ ֆունկցիան՝ որպես արժեք, որը կարելի է պահել փոփոխականում ու փոխանցել այնպես, ինչպես անանուն ֆունկցիաները.

```elixir
iex> len = &String.length/1
&String.length/1
iex> len.("hello")
5
iex> is_function(len, 1)
true
```

Փաստացի `&String.length/1`-ը «բռնում է» արդեն գոյություն ունեցող `String.length/1` ֆունկցիան եւ հանձնում մեզ որպես սովորական արժեք։ Այն կանչում ենք կետով՝ `len.("hello")`, եւ `is_function/2`-ը հաստատում է, որ ունենք իսկապես մեկ արգումենտ սպասող ֆունկցիա։

Սա կոչվում է capture՝ «բռնում»։ Բռնված ֆունկցիան կանչվում է կետով և փոխանցվում է ուր ուզեք։ Kernel-ի ֆունկցիաների համար մոդուլի անունը կարելի է բաց թողնել, իսկ սեփական մոդուլի ներսում կարելի է բռնել նույնիսկ private ֆունկցիա.

```elixir
iex> fun = &is_atom/1
&:erlang.is_atom/1
iex> fun.(:hello)
true
iex> fun.(123)
false
```

Իսկ հիմա մի փաստ. Elixir-ում **օպերատորներն էլ են ֆունկցիաներ**։ `+`-ը պարզապես `+/2` ֆունկցիան է, ուրեմն նրան էլ կարելի է բռնել.

```elixir
iex> add = &+/2
&:erlang.+/2
iex> add.(1, 2)
3
```

### &֊ը որպես կարճ գրելաձև

`&`-ը նաև օգտագործվում է նոր անանուն ֆունկցիաներ ստեղծելու համար՝ առանց `fn`-ի ու `end`-ի։ Արգումենտների տեղում գրվում են `&1`, `&2`, ... տեղապահները.

```elixir
iex> double = &(&1 * 2)
#Function<42.3316493/1 in :erl_eval.expr/6>
iex> double.(21)
42
iex> greet = &("Good " <> &1)
#Function<42.3316493/1 in :erl_eval.expr/6>
iex> greet.("morning")
"Good morning"
```

`&(&1 * 2)`-ը բառացիորեն նույն `fn x -> x * 2 end`-ն է. ոչ ավել, ոչ պակաս։ Երկու կանոն. տեղապահները սկսվում են `&1`-ից, և այս գրելաձևով **առանց արգումենտի** ֆունկցիա ստեղծել չի կարելի՝ գոնե մեկ արգումենտ պետք է լինի։

Ե՞րբ որն օգտագործել։ Փորձված խորհուրդ. `&`-ը հիանալի է մեկտողանոց, ակնհայտ պարզ ֆունկցիաների համար, իսկ հենց տրամաբանությունը մի քիչ երկարում է՝ `fn`-ն ավելի ընթեռնելի է։ `&(&1 + &2 * &1 - &3)` գրողին ոչ ոք շնորհակալություն չի ասի։

### Pattern matching-ը ֆունկցիայի արգումենտներում

[Անանուն ֆունկցիաների](./anonymous_functions.md) ճյուղավորումը հիշո՞ւմ եք։ Այժմ լավ նորություն. **նույն մեխանիզմն աշխատում է նաև մոդուլային ֆունկցիաներում**։ Նույն անունով ու arity-ով մի քանի `def` իրար տակ գրելիս ստանում ենք ճյուղեր. Elixir-ը վերևից ներքև գտնում է առաջին կաղապարը, որին համընկնում է արգումենտը.

```elixir
iex> defmodule Status do
...>   def describe(200), do: "OK"
...>   def describe(404), do: "not found"
...>   def describe(_code), do: "unknown status"
...> end
iex> Status.describe(200)
"OK"
iex> Status.describe(500)
"unknown status"
```

Ամեն ճյուղում արգումենտը կաղապար է՝ կա՛մ կոնկրետ արժեք (`200`), կա՛մ underscore-ով սկսվող անուն (`_code`)։ Այն ամենը, ինչ սովորեցինք [pattern matching-ի հոդվածում](./pattern_matching.md), գործում է նաև այստեղ։ Կարևոր է ճյուղերի հերթականությունը. եթե ամենավերջին՝ բոլորին համընկնող ճյուղը (`_code`-ով) տեղափոխես վերև, մյուսների հերթը այլևս երբեք չի հասնի, և կոմպիլյատորը հենց դրա մասին կզգուշացնի՝ «this clause cannot match because a previous clause always matches»։ Եվ ընդհանրապես կոմպիլյատորի զգուշացումները երբեք պետք չէ անտեսել. երբեմն դրանք կոսմետիկ բնույթի վրիպակների մասին են, բայց ընդհանուր առմամբ կոմպիլյատորի զգուշացումներն անտեսելը կարող է բերել համակարգային սխալների։

Իսկ tuple-ների վերաբերյալ ծանոթ հարցը՝ `{:ok, ...}`-ի ու `{:error, ...}`-ի հետ ինչ անել, մոդուլային ֆունկցիաներում ստանում է էլեգանտ լուծում.

```elixir
iex> defmodule Reader do
...>   def show({:ok, contents}), do: contents
...>   def show({:error, _reason}), do: "could not read the file"
...> end
iex> Reader.show(File.read("missing.txt"))
"could not read the file"
```

Ֆունկցիայի ներսում ոչ մի `if`, ոչ մի ստուգում. երկու հնարավոր ելքը դարձել է երկու առանձին, կարճ ու պարզ ֆունկցիա։ Հենց սա է Elixir-ի կոդի ամենաճանաչելի դիմագիծը. ֆունկցիաները սովորաբար շատ են ու փոքր, որովհետև ո՛ր ճյուղով գնալը որոշում են հենց կաղապարները, ոչ թե ձեռքով գրված պայմանները։

### Guard-եր. պայման կաղապարի կողքին

Կաղապարը ստուգում է արժեքի **կառուցվածքը**, բայց երբեմն դա բավական չէ՝ պետք է ստուգել նաև արժեքի **հատկությունը**. օրինակ թիվ է, բայց արդյոք դրակա՞ն է։ Հենց այստեղ են օգնության հասնում guard-երը՝ `when` բառով ավելացվող պայմանները.

```elixir
iex> defmodule Sign do
...>   def of(0), do: :zero
...>   def of(x) when x > 0, do: :positive
...>   def of(_x), do: :negative
...> end
iex> Sign.of(7)
:positive
iex> Sign.of(-7)
:negative
iex> Sign.of(0)
:zero
```

Այս մեկ օրինակում իրար կողքի աշխատում են երեք տարբերակ՝ կոնկրետ արժեք (`0`), կաղապար ու guard միասին (`x when x > 0`), և բոլորին համընկնող ճյուղը (`_x`)։ Ճյուղն ընտրվում է միայն այն ժամանակ, երբ թե՛ կաղապարն է համընկնում, թե՛ guard-ը `true` է վերադարձնում. այլապես Elixir-ը պարզապես անցնում է հաջորդին։

Նույն `when`-ով guard-եր կարելի է ավելացնել նաև անանուն ֆունկցիաների ճյուղերին։ Կա, սակայն, մեկ կարևոր սահմանափակում. guard-ի ներսում ամեն բան չէ, որ թույլատրվում է։ Կարելի է օգտագործել համեմատություններ (`>`, `==` և այլն), թվաբանություն և տիպը ստուգող ֆունկցիաներ (մեզ ծանոթ `is_number/1`, `is_atom/1`, `is_function/1` և այլն․․․․)։ Իսկ մեր ստեղծած սովորական ֆունկցիաները guard-երում կանչել չենք կարող, և դա պատահական չէ. guard-ը պետք է աշխատի ակնթարթորեն ու առանց կողմնակի հետևանքների, քանի որ այն ստուգվում է հենց ճյուղն ընտրելու պահին։

### Ի՞նչ է սպասվում առջևում

Ֆունկցիաների գործիքակազմն այժմ ամբողջական է. մեր ձեռքի տակ են և՛ մոդուլային, և՛ անանուն ֆունկցիաները, `&` օպերատորը, ինչպես նաև կաղապարներով ու guard-երով ճյուղերը։ Հաջորդիվ կուսումնասիրենք `Enum` մոդուլը, և այս ամբողջ գիտելիքը վերջապես կիրառության մեջ կդնենք հավաքածուների վրա։
