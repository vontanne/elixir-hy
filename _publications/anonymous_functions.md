## Անանուն ֆունկցիաներ

[Նախորդ հոդվածի](./modules_and_functions.md) ֆունկցիաներից ամեն մեկն ուներ անուն և պարփակված էր իր մոդուլում։ Այսօր կխոսենք բոլորովին այլ՝ **անանուն ֆունկցիաներ**֊ի մասին, որոնք գոյություն ունեն մոդուլից դուրս, պահվում են փոփոխականների մեջ և փոխանցվում այնպես, ինչպես թիվը կամ string-ը։ Հենց սա է գլխավոր միտքը, որին արժի ուշադրություն դարձնել. **ֆունկցիաները Elixir-ում կարող են հանդես գալ որպես սովորական արժեք**։

JavaScript-ից եկածների համար այստեղ ընդհանրապես ոչ մի տարօրինակ բան չկա. `fn a, b -> a + b end`-ը նույն `(a, b) => a + b`-ն է՝ մի քիչ տարբերվող սինթաքսով։ Java-ի ու C#-ի հիմք ունեցող ծրագրավորողների համար ամենահասկանալին lambda-ների հետ համեմատությունն է։

### fn-ից մինչև առաջին կանչը

Անանուն ֆունկցիան սահմանվում է `fn ... end` բանալի բառերով. `->` սլաքից ձախ արգումենտներն են, աջ՝ մարմինը.

```elixir
iex> add = fn a, b -> a + b end
#Function<42.3316493/2 in :erl_eval.expr/6>
iex> add.(1, 2)
3
```

Այստեղ երկու նորություն կա։ Առաջին՝ iex-ի պատասխանը. ֆունկցիան արժեք է, բայց «ցուցադրելու բան» չունի, դրա համար iex-ը տպում է ծառայողական `#Function<...>` նշումը (թվերը ձեզ մոտ այլ կլինեն, իսկ `:erl_eval` կտորը պարզապես ասում է, որ ֆունկցիան սահմանվել է shell-ում)։ Ուշադրություն դարձրեք `/2`-ին այդ նշման մեջ. անանուն ֆունկցիաներն էլ ունեն arity։

Երկրորդ նորությունը կանչի **կետն** է՝ `add.(1, 2)`։

### Ինչո՞ւ կետ

Պատճառը հասկանալու համար նախ պետք է իմանալ մի փաստ, Elixir-ում **ֆունկցիաների անուններն ու փոփոխականների անունները պահվում են տարբեր namespace-ներում**։ `def`-ով սահմանված ֆունկցիան գրանցվում է համապատասխան մոդուլի ֆունկցիաների namespace-ում։ Իսկ անանուն ֆունկցիան արժեք է, հետևաբար ապրում է փոփոխականի մեջ՝ բոլորովին այլ namespace֊ում։

Երկու namespace՝ կանչի երկու տարբեր գրելաձև։ `add(1, 2)` գրելաձևը կոմպիլյատորի համար ունի ուղիղ մեկ իմաստ. «գտի՛ր `add/2` **անունով** ֆունկցիան՝ այս մոդուլում կամ ներմուծվածների մեջ, և կանչի՛ր»։ Փոփոխականներին այս գրելաձևը **երբեք** չի նայում։ Իսկ `add.(1, 2)`-ը ճիշտ հակառակն է. «վերցրո՛ւ `add` փոփոխականի **արժեքը** և կանչի՛ր այն որպես ֆունկցիա»։

Քանի որ namespace-ները տարբեր են, նույն անունը երկուսում միաժամանակ կարող է գոյություն ունենալ։ Փորձենք հետևյալ օրինակը՝ ստեղծենք անանուն ֆունկցիա, և կապենք այն `length` անունով փոփոխականի հետ, երբ կա նաև ներկառուցված `length/1`-ը.(հուսով եմ հիշում եք, որ այստեղ `=` օպերատորը վերագրման օպերատոր չէ, ինչպես մենք սովոր ենք մեյնսթրիմային ծրագրավորման լեզուներում, և մենք անանուն ֆունկցիան որպես արժեք ոչ թե վերագրում ենք length փոփոխականին, այլ ընդամենը կապում արժեքը նրա հետ)

```elixir
iex> length = fn _list -> :fake end
#Function<42.3316493/1 in :erl_eval.expr/6>
iex> length([1, 2, 3])
3
iex> length.([1, 2, 3])
:fake
```

Ինչպես տեսնում ենք՝ ոչ մի կոնֆլիկտ և ոչ մի սխալ։ Փակագծերով կանչը փնտրեց ֆունկցիաների namespace-ում և գտավ `Kernel`-ի `length/1`-ը՝ և ֆունկցիայի կանչի արդյունքը վերադարձրեց 3։ Իսկ կետով կանչը փոփոխականների namespace֊ից գտավ և կանչեց մեր անանուն ֆունկցիան՝ վերադարձնելով `:fake`։ Կետը հենց այդ ընտրության մասին է. **առանց կետի՝ անունով սահմանված ֆունկցիա, կետով՝ փոփոխականի մեջ պահված արժեք**։

### Ֆունկցիան որպես արժեք

Եթե ֆունկցիան արժեք է, ուրեմն նրա հետ կարելի է վարվել ինչպես արժեքի հետ. ստուգել տիպը, փոխանցել այլ ֆունկցիայի որպես արգումենտ և այլն։ Ստուգելու համար կան `is_function/1`-ը և `is_function/2`-ը (երկրորդը ստուգում է նաև arity-ն).

```elixir
iex> is_function(add)
true
iex> is_function(add, 2)
true
iex> is_function(add, 1)
false
```

Այժմ ֆունկցիաները որպես արգումենտ օգտագործելու մասին։ `Enum` մոդուլի ֆունկցիաները աշխատում են հենց այսպես. նրանց տալիս ես հավաքածու **և ֆունկցիա**, նրանք այդ ֆունկցիան կիրառում են ամեն տարրի վրա։

```elixir
iex> Enum.map([1, 2, 3], fn x -> x * 2 end)
[2, 4, 6]
```

Չափազանց պարզ է ու ընթեռնելի. «ամեն `x`-ի համար՝ կատարիր `x * 2` գործողությունը»։ Սա մի փոքր օրինակ է. `Enum`-ի ֆունկցիաների հետ ավելի մանրամասն կծանոթանանք առաջիկայում։

### Closure-ներ. ֆունկցիան «հիշում» է իր միջավայրը

Անանուն ֆունկցիան «տեսնում» է այն փոփոխականները, որոնք գոյություն ունեին իր սահմանման պահին.

```elixir
iex> greeting = "Hello"
"Hello"
iex> greet = fn name -> greeting <> ", " <> name end
#Function<42.3316493/1 in :erl_eval.expr/6>
iex> greet.("Ani")
"Hello, Ani"
```

Սա կոչվում է closure, և JS֊ին ծանոթ ծրագրավորողների համար սա ամբողջովին հասկանալի տերմին է։ Բայց այստեղ թաքնված է մի կարևոր տարբերություն, որի վրա JS-ից եկածները հաճախ սայթաքում են։ JavaScript-ում closure-ի ներսից կարելի է **փոխել** դրսի փոփոխականը։ Elixir-ում՝ երբեք.

```elixir
iex> x = 42
42
iex> (fn -> x = 0 end).()
0
iex> x
42
```

Ֆունկցիայի մարմնում `x = 0` արտահայտությունն ընդամենը ստեղծեց **նոր, ներքին** `x`, որը վերացավ երբ ֆունկցիայի աշխատանքն ավարտվեց։ Դրսի `x`-ը մնաց 42։ Ոչ մի ֆունկցիա, ոչ մի տեղից, չի կարող փոխել ձեր փոփոխականը. տվյալների անփոփոխելիությունը, որին հանդիպել ենք բոլոր տիպերի մոտ, այստեղ էլ է գործում։

### Մի քանի կաղապար՝ մեկ ֆունկցիայում

Հիմա դիտարկենք երկու կարևոր թեմա։ Անանուն ֆունկցիայի արգումենտներն իրականում [կաղապարներ](./pattern_matching.md) են, և ֆունկցիան կարող է ունենալ մի քանի clause՝ «ճյուղ». Elixir-ը վերևից ներքև գտնում է առաջին համընկնող կաղապարը և կատարում հենց այդ ճյուղը։ Շեշտենք, որ սա անանուն ֆունկցիաների հատկություն չէ միայն. ճիշտ նույն մեխանիզմը՝ կաղապարներ արգումենտներում և մի քանի clause, ունեն նաև մոդուլային, named ֆունկցիաները, ուղղակի անանուն ֆունկցիաներում դրանք ավելի էլեգանտ են նայվում։

```elixir
iex> negate = fn
...>   true -> false
...>   false -> true
...> end
iex> negate.(false)
true
```

Իսկ հիմա հիշեք tuple-ների մասին հոդվածը. ինչպե՞ս համառոտ կերպով մշակել և՛ `{:ok, ...}`-ի, և՛ `{:error, ...}`-ի տարբերակները․

```elixir
iex> handler = fn
...>   {:ok, contents} -> contents
...>   {:error, _reason} -> "could not read the file"
...> end
iex> handler.(File.read("missing.txt"))
"could not read the file"
```

Հաջողության դեպքում կաղապարը հանում է պարունակությունը, ձախողման դեպքում `_reason`-ը: Ոչ մի `if`, ոչ մի ձեռքով ստուգում:

Երկու կանոն այս ճյուղերի մասին. բոլոր clause-ները պետք է ունենան **նույն arity-ն** (տարբեր քանակի արգումենտներով ճյուղերը Elixir-ը մերժում է դեռ կոմպիլյացիայի պահին), և եթե արժեքը ոչ մի կաղապարի չհամընկնի, կստանաք `FunctionClauseError`։ Իսկ այդ ճյուղերին կարելի է կցել նաև լրացուցիչ պայմաններ՝ guard-եր. դրանք առանձին զրույցի թեմա են, կխոսենք հետագայում։

### Հարմար գործիք ձեր գրպանում. h

Նախորդ հոդվածից գիտենք, որ ֆունկցիայի լրիվ «անուն-ազգանունը» `անուն/arity` ձևն է։ iex-ում `h` հրամանը անմիջապես տերմինալում ցույց է տալիս ցանկացած ֆունկցիայի պաշտոնական նկարագրությունը.

```elixir
iex(1)> h Map.get/3

def get(map, key, default \\ nil)                        
@spec get(map(), key(), value()) :: value()
Gets the value for a specific key in map.

If key is present in map then its value value is returned. Otherwise, default
is returned.

If default is not provided, nil is used.

## Examples

    iex> Map.get(%{}, :a)
    nil
    iex> Map.get(%{a: 1}, :a)
    1
    iex> Map.get(%{a: 1}, :b)
    nil
    iex> Map.get(%{a: 1}, :b, 3)
    3
    iex> Map.get(%{a: nil}, :a, 1)
    nil

iex(2)> 
```

Փորձեք նաև `h String.length/1` կամ պարզապես `h`՝ առանց արգումենտի։ Այս մեկ տառը ձեր վստահելի օգնականն է դառնալու։

### Փորձեք ինքներդ

Բացեք iex-ը և սահմանեք `square` անանուն ֆունկցիա, որը թիվը բազմապատկում է ինքն իրենով։ Նախ կանչեք այն **առանց** կետի՝ `square(3)`, և ուշադիր կարդացեք սխալը. Elixir-ը ո՞ր ֆունկցիան է փնտրում և չի գտնում։ Ապա կանչեք ճիշտ ձևով։ Ստուգեք `is_function(square, 1)`-ը և `is_function(square, 2)`-ը։ Կրկնեք closure-ի փորձը. `x = 42`, ֆունկցիայի ներսում `x = 0`, և համոզվեք, որ դրսի `x`-ը անփոփոխ է։ Վերջում գրեք երկու ճյուղանի `describe` ֆունկցիա, որը `{:ok, _}`-ի դեպքում վերադարձնում է `"success"`, իսկ `{:error, _}`-ի դեպքում՝ `"failure"`, և կանչեք նրան `File.read`-ի իրական պատասխաններով՝ մեկ գոյություն ունեցող, մեկ էլ գոյություն չունեցող ֆայլով։
