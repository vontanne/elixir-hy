## Pattern matching-ը Elixir-ում

Վերջին հոդվածներում pattern matching-ը մեզ ուղեկցում էր ամենուր. [bitstring-երի վարժություններից մեկում](./bitstrings_and_binaries.md) գույնը բաժանեցինք `<<r, g, b>>`-ով, [map-երից](./maps.md) արժեքներ հանեցինք հատուկ գրելաձևով, նույնը՝ [tuple-երի մասին հոդվածում](./tuples.md) բացեցինք `{:ok, ...}`-ը օգտագործելով կաղապարներ։ Եկել է խոստացված պահը՝ հավաքել այդ կտորները մեկ ամբողջական պատկերի մեջ։

Եթե կարճ՝ pattern matching-ը երկու գործ է անում միաժամանակ՝ **ստուգում է** արժեքի ձևը և **հանում է** դրանից մեզ պետք եղածը։ Հենց այս զույգն է այն դարձնում Elixir-ի ամենասիրված գործիքներից մեկը․

### «=» նշանի իսկական դերը

Սկսենք ամենասկզբից։ Elixir-ում `=`-ը կոչվում է match operator՝ համապատասխանեցման օպերատոր, և վերագրման նշան չէ.

```elixir
iex> x = 1
1
iex> 1 = x
1
```

Կանգ առեք երկրորդ տողի վրա։ Օրինակ JavaScript-ում `1 = x` գրելը շարահյուսական սխալ է. թվային լիթերալին բան չես վերագրի։ Elixir-ում սա սովորական տող է. ձախ կողմում կաղապարն է (թիվ 1-ը), աջում՝ արժեքը (`x`-ը, որը 1 է), եթե դրանք համընկնում են, ամեն ինչ կարգին է։ Իսկ երբ չեն համընկնում.

```elixir
iex> 2 = x
** (MatchError) no match of right hand side value: 1
```

Կողմերն իրավահավասար չեն. **ձախում միշտ կաղապարն է, աջում՝ արժեքը**։ Աջ կողմը սովորական Elixir կոդ է և այն պետք է արժեք ունենա. եթե `y`-ը երբեք արժեք չի ստացել, `1 = y` գրելիս սխալ կստանաք, ծրագիրն ուղղակի կոմպիլյացիա չի լինի։

Կաղապարում կարող են լինել ցանկացած literal արժեքներ՝ թվեր, atom-ներ, string-եր, եթե դրանք համընկնում են միայն ինքներն իրենց հետ.

```elixir
iex> :ok = :ok
:ok
iex> :ok = :error
** (MatchError) no match of right hand side value: :error
iex> 1 = 1.0
** (MatchError) no match of right hand side value: 1.0
```

Վերջին տողը ծանոթ պատմություն է. ինչպես map-ի բանալիների դեպքում, այստեղ էլ համեմատությունը խիստ է՝ ամբողջ թիվը կոտորակայինի-ի հետ չի համընկնում։

### Rebinding-ը և pin օպերատորը (^)

Փոփոխականը կաղապարում միշտ նոր արժեք է ստանում, նույնիսկ եթե արդեն ուներ.

```elixir
iex> x = 1
1
iex> x = 2
2
```

Սա կոչվում է rebinding՝ անունը կապվում է նոր արժեքի հետ։ Բայց երբեմն ուզում ենք ճիշտ հակառակը. ոչ թե `x`-ին նոր արժեք տալ, այլ **ստուգել**, որ եկած արժեքը հավասար է `x`-ի եղած արժեքին։ Դրա համար կա pin օպերատորը՝ `^` նշանը.

```elixir
iex> x = 1
1
iex> ^x = 1
1
iex> ^x = 2
** (MatchError) no match of right hand side value: 2
```

`^x`-ը կաղապարում նշանակում է «այստեղ պետք է լինի հենց `x`-ի ներկայիս արժեքը»։

```elixir
iex> expected = :ok
:ok
iex> {^expected, value} = {:ok, 42}
{:ok, 42}
iex> value
42
```

Եվս մեկ նրբություն. եթե նույն անունը կաղապարում հանդիպում է երկու անգամ, երկու տեղում էլ պետք է նույն արժեքը լինի.

```elixir
iex> {x, x} = {1, 1}
{1, 1}
iex> {x, x} = {1, 2}
** (MatchError) no match of right hand side value: {1, 2}
```

Այսինքն՝ `{x, x}` կաղապարը կարդացվում է «երկու տարր, որոնք իրար հավասար են»։

### «_»՝ «սա ինձ չի հետաքրքրում»

Հաճախ կաղապարից պետք է միայն մի մասը, իսկ մնացածը՝ ինչ ուզում է, թող լինի։ Դրա համար կա underscore-ը՝ `_` նշանը, որը ամեն ինչի հետ համընկնում է.

```elixir
iex> {_, amount} = {:not_important, 500}
{:not_important, 500}
iex> amount
500
```

`_`-ի արժեքը կարդալ չի կարելի. այն «սև անցք» է, ինչ ընկավ մեջը՝ կորում է։ Գործնականում հաճախ կտեսնեք նաև `_reason`, `_name` ձևի անուններ. underscore-ով սկսվող փոփոխականն աշխատում է սովորականի պես, բայց կոդը ընթերցողին (և Elixir-ի կոմպիլյատորին) ասում է՝ «այս արժեքը գիտակցաբար չենք օգտագործում»։

### List-երի կաղապարները և «|» նշանի վերադարձը

List-ը կաղապարում իրեն պահում է tuple-ի պես. տարր առ տարր պետք է համընկնում լինի.

```elixir
iex> [a, b, c] = [10, 20, 30]
[10, 20, 30]
iex> b
20
```

Բայց հիշենք list-երի հոդվածը. list-ի բնական ձևը «գլուխ + պոչ» կառուցվածքն է, և տարր ավելացնում էինք `[0 | list]` գրելաձևով։ Պարզվում է նույն `|` նշանը կաղապարում աշխատում է **հակառակ ուղղությամբ**՝ list-ը բաժանում է գլխի և պոչի.

```elixir
iex> [head | tail] = [1, 2, 3]
[1, 2, 3]
iex> head
1
iex> tail
[2, 3]
```

Ծանո՞թ է։ Սա ճիշտ այն է, ինչ անում էին `hd/1`-ը և `tl/1` ֆունկցիաները։ Կարելի է անջատել նաև մեկից ավելի տարր.

```elixir
iex> [first, second | rest] = [1, 2, 3, 4]
[1, 2, 3, 4]
iex> rest
[3, 4]
```

Ուշադրություն. `[head | tail]`-ը պահանջում է **առնվազն մեկ տարր**, դատարկ list-ի հետ այն չի աշխատում.

```elixir
iex> [head | tail] = []
** (MatchError) no match of right hand side value: []
```

Տրամաբանական է. դատարկ list-ը գլուխ չունի, որ անջատես։

### Tuple և map

Tuple-երի ու map-երի կաղապարներն արդեն գիտենք, ուստի միայն ամրապնդենք գլխավոր հակադրությունը.

- **Tuple-ի կաղապարը խիստ է.** պետք է նկարագրի ամբողջ tuple-ը՝ ճիշտ նույն քանակի տարրերով։
- **Map-ի կաղապարն ազատ է.** նշում ենք միայն մեզ պետք եղած բանալիները։

Այս տարբերությունը ամենապարզ երևում է «դատարկ» կաղապարների վրա։ Դատարկ `[]`-ը համընկնում է միայն դատարկ list-ի հետ, դատարկ `{}`-ը՝ միայն դատարկ tuple-ի։ Իսկ դատարկ `%{}`-ը ոչ մի բանալի չի պահանջում, ուրեմն համընկնում է **ցանկացած** map-ի հետ.

```elixir
iex> %{} = %{name: "Ani"}
%{name: "Ani"}
iex> [] = [1]
** (MatchError) no match of right hand side value: [1]
iex> {} = {1}
** (MatchError) no match of right hand side value: {1}
```

### String-ների ու binary-ների կաղապարները

Խոսենք նաև bitstring-երից։ Binary-ն կաղապարում գրվում է նույն `<<>>` փակագծերով, և ամեն անուն «վերցնում» է մեկական byte.

```elixir
iex> <<r, g, b>> = <<255, 165, 0>>
<<255, 165, 0>>
iex> g
165
```

Իսկ string-երի համար (որոնք, հիշում ենք, UTF-8 binary-ներ են) կա շատ գործնական մի հնարք՝ `<>` օպերատորի օգտագործմամբ.

```elixir
iex> "hello " <> rest = "hello world"
"hello world"
iex> rest
"world"
```

Կաղապարը ստուգում է, որ string-ը սկսվում է `"hello "`-ով, և մնացածը դնում է `rest`-ի մեջ։ Ուշադրություն. այսպես աշխատում է միայն **սկզբնամասը**. `something <> " world"` ձևի կաղապար (վերջնամասով) Elixir-ը թույլ չի տալիս։

### Կաղապար կաղապարի մեջ

Եվ ամենագեղեցիկը. կաղապարներն ազատ օգտագործվում են իրար մեջ։ Իրական ծրագրերում ֆունկցիաները հաճախ վերադարձնում են «tuple, որի մեջ map կա», և մեկ տողով կարող ենք և՛ ստուգել հաջողությունը, և՛ map-ից հանել կոնկրետ արժեքը.

```elixir
iex> response = {:ok, %{name: "Hasmik", age: 25}}
{:ok, %{age: 25, name: "Hasmik"}}
iex> {:ok, %{name: name}} = response
{:ok, %{age: 25, name: "Hasmik"}}
iex> name
"Hasmik"
```

Մեկ տող՝ երեք գործ. ստուգեցինք, որ արդյունքը `:ok` է, որ ներսում map կա `name` բանալիով, և հանեցինք արժեքը։ Հենց այս է pattern matching-ի ուժը։

### Փորձեք ինքներդ

Բացեք iex-ը։ Ստեղծեք `x = 10`, ապա փորձեք `10 = x`, `^x = 10` և `^x = 20`՝ կանխագուշակելով ամեն տողի արդյունքը նախքան կոդը աշխատացնելը։ Ստեղծեք հինգ տարրանոց list և `[first, second | rest]` կաղապարով անջատեք առաջին երկուսը. ապա նույն կաղապարը փորձեք մեկ տարրանոց list-ի վրա և բացատրեք ստացված արդյունքը։ Ստեղծեք `"Elixir is fun"` string-ը և `<>`-ով անջատեք `"Elixir "` սկզբնամասից հետո եկող մասը։ Վերջում ստեղծեք `{:error, %{code: 404, message: "not found"}}` արժեքը և մեկ կաղապարով հանեք միայն `code`-ը՝ underscore-ով անտեսելով այն, ինչ պետք չէ։

### Ի՞նչ է սպասվում առջևում

Այս հոդվածում կաղապարներն աշխատում էին միայն `=` նշանի հետ։ Սակայն Elixir-ում կաղապարներ են գրվում նաև ֆունկցիաների արգումենտներում և `case` կառուցվածքում, որտեղ ծրագիրն ինքն է ընտրում, թե որ կաղապարն է համընկնում։ Իսկ երբ կաղապարին պետք է ավելացնել լրացուցիչ պայմաններ, օգնության են գալիս guard-երը։ Այս ամենի մասին՝ առաջիկա հոդվածներում։
