## Ծանոթություն Enum մոդուլի հետ

`Enum`-ը Elixir-ի ամենից հաճախ կիրառվող մոդուլներից է։ Այն ծրագրավորողները, ովքեր ծանոթ են JavaScript֊ին, զանգվածների հետ աշխատելիս սովոր են դիմել `.map`-ին, `.filter`-ին և `.forEach`-ին։ Elixir-ում այս մոտեցումը ոչ թե ընտրություններից մեկն է, այլ հիմնականը, որովհետև մեր իմացած ցիկլերն այստեղ գոյություն չունեն։ Կա `for` կառուցվածք, որը որոշակի նմանություն ունի ավանդական for ցիկլի հետ, սակայն քանի որ Elixir-ը ֆունկցիոնալ լեզու է և տվյալներն անփոփոխ են, իրական ցիկլ ստեղծել հնարավոր չէ։ Այդ `for`-ն ընդամենը մակրո է, որի մասին կխոսենք հետագայում։

Թեման ծավալուն է, ուստի այն ներկայացնելու ենք մի քանի հոդվածով։ 

### Ի՞նչ է enumerable-ը

`Enum`-ի ֆունկցիաներն աշխատում են enumerable-ների հետ՝ այն ամենի, ինչի տարրերի վրայով հնարավոր է հաջորդաբար անցնել։ Դրանցից երկուսը մեզ արդեն ծանոթ են՝ [list](./lists.md)-երն ու [map](./maps.md)-երը։ Երրորդին ժամանակն է ծանոթանալու. դա **Range**-ն է։

Range-ը թվերի միջակայք է, որը գրվում է երկու կետով.

```elixir
iex> 1..5
1..5
iex> Enum.to_list(1..5)
[1, 2, 3, 4, 5]
```

Այստեղ կա մի կարևոր նրբություն։ `1..5`-ը հնգանդամ list **չի** ստեղծում. այն պահում է ընդամենը երկու թիվ՝ սկիզբը և վերջը։ Միջանկյալ տարրերն առաջանում են միայն այն պահին, երբ որևէ ֆունկցիա անցնում է միջակայքի վրայով։ Այս պատճառով `1..1_000_000` գրելը նույնքան քիչ ռեսուրս է պահանջում, որքան `1..5`-ը։ Հենց այս հատկության շնորհիվ Range-ը `Enum` ուսումնասիրելու համար ամենահարմար տիպն է, և այս հոդվածաշարում այն հաճախ է հանդիպելու։

### map. յուրաքանչյուր տարրի ձևափոխումը

`Enum.map/2`-ին [արդեն հանդիպել ենք](./anonymous_functions.md)։ Այն որպես արգումենտ ստանում է list և ֆունկցիա, ֆունկցիան կիրառում է ամեն տարրի վրա, իսկ արդյունքները հավաքում **նոր** list-ում.

```elixir
iex> Enum.map([1, 2, 3], fn x -> x * 2 end)
[2, 4, 6]
iex> Enum.map(1..3, fn x -> x * 2 end)
[2, 4, 6]
```

Հարց է առաջանում՝ ի՞նչ է կատարվում, երբ `map`-ին փոխանցում ենք ոչ թե list, այլ map հավաքածու։ Այս դեպքում ամեն բանալի-արժեք զույգը ֆունկցիային է հասնում որպես երկտարր tuple, և pattern matching-ի օգնությամբ հենց արգումենտում բանալին և իր համապատասխան արժեքն առանձնացվում է.

```elixir
iex> Enum.map(%{"a" => 1, "b" => 2}, fn {k, v} -> {k, v * 2} end)
[{"a", 2}, {"b", 4}]
```

Ուշադրություն դարձրեք արդյունքին. ստացանք ոչ թե map, այլ tuple-ների **list**։ Ինչ տիպի հավաքածու էլ փոխանցվի `Enum`-ին, արդյունքը գրեթե միշտ list է։ Թե ինչպես արդյունքը կրկին map դարձնել, կխոսենք հետագայում։

### filter և reject. տարրերի ընտրությունը

`Enum.filter/2`-ը պահում է միայն այն տարրերը, որոնց համար ֆունկցիան դրական պատասխան է վերադարձնում.

```elixir
iex> Enum.filter([1, 2, 3, 4, 5], fn x -> rem(x, 2) == 0 end)
[2, 4]
```

(`rem/2`-ը բաժանման մնացորդն է՝ մեզ ծանոթ FizzBuzz-ի օրինակից. եթե մնացորդը 0 է, ուրեմն թիվը զույգ է։)

Սակայն ի՞նչ է նշանակում «դրական պատասխան» կամ «բացասական պատասխան»։ Այստեղ գործում է մի կարևոր պայմանավորվածություն, որը վերաբերում է ամբողջ Elixir-ին. «բացասական պատասխան» (falsy) են համարվում միայն `false`-ը և `nil`-ը, մնացած բոլոր արժեքները՝ «դրական պատասխան» են (truthy): Անգամ 0-ն, դատարկ string-ը, դատարկ list-ը՝ բոլորն էլ դրական են համարվում։

`Enum.reject/2`-ը `filter`-ի հակառակ գործողությունն է. այն հեռացնում է այն տարրերը, որոնց համար ֆունկցիան «դրական պատասխան» (truthy) է վերադարձնում.

```elixir
iex> Enum.reject([1, 2, 3, 4, 5], fn x -> rem(x, 2) == 0 end)
[1, 3, 5]
```

### each. գործողություն՝ առանց նոր հավաքածուի

`Enum.each/2`-ը նույնպես անցնում է հավաքածուի բոլոր տարրերի վրայով, սակայն նրա նպատակն այլ է։ Այն նոր հավաքածու չի կառուցում, այլ ամեն տարրի հետ կատարում է որևէ գործողություն, օրինակ արտածում է տերմինալում։

```elixir
iex> Enum.each(["some", "example"], fn x -> IO.puts(x) end)
some
example
:ok
```

Ֆունկցիան վերջում վերադարձնում է `:ok`։ `each`-ը գործողության արդյունք չի հավաքում, այլ պարզապես հաստատում է, որ գործողությունը կատարվել է։ Կիրառության սկզբունքը հետևյալն է. եթե մեզ անհրաժեշտ է արդյունքը, կիրառվում է `map`, իսկ եթե միայն գործողությունը՝ `each`։ Սա համապատասխանում է JavaScript-ի `.map`-ի և `.forEach`-ի տարբերությանը։

### Երեք հաճախակի կիրառվող ֆունկցիա. sum, count, member?

Ստորև ներկայացված երեք ֆունկցիաները հաճախ կիրառվում են ամենօրյա աշխատանքում։ `Enum.sum/1`-ը գումարում է հավաքածուի բոլոր տարրերը (այն մեզ ծանոթ է list-երի հոդվածից, սակայն այժմ գիտենք, որ այն կիրառելի է նաև Range-ի վրա).

```elixir
iex> Enum.sum(1..100)
5050
```

`Enum.count/1`-ը հաշվում է տարրերի քանակը, իսկ երկու արգումենտով տարբերակը՝ միայն տրված պայմանին բավարարող տարրերը.

```elixir
iex> Enum.count([1, 2, 3, 4, 5])
5
iex> Enum.count([1, 2, 3, 4, 5], fn x -> rem(x, 2) == 0 end)
2
```

`Enum.member?/2`-ը ստուգում է՝ արդյոք տվյալ տարրը առկա է հավաքածույում։ Կարևոր է նշել, որ ստուգումը **խիստ** է, կատարվում է `===` օպերատորով, ինչպես map-ի բանալիների և pattern matching-ի դեպքում.

```elixir
iex> Enum.member?(1..10, 5)
true
iex> Enum.member?(1..10, 5.0)
false
```

Ամբողջ 5-ը միջակայքում առկա է, իսկ float 5.0-ը՝ ոչ։ Elixir-ը տիպերը երբեք ինքնաշխատ չի ձևափոխում, և այս կարևոր սկզբունքը արդեն պետք է որ ամրապնդված լինի ընթերցողի մոտ։

### Մի քանի խոսք արդյունավետության մասին

`Enum`-ի ֆունկցիաներն աշխատում են գծային բարդությամբ՝ O(n), ինչպես ձևակերպել ենք list-երի հոդվածում։ Սա տրամաբանական է. բոլոր տարրերի վրայով անցնող ֆունկցիան բնականաբար չի կարող ավելի արդյունավետ՝ հաստատուն կամ լոգարիթմական բարդությամբ աշխատել։ Սա թերություն չէ, այլ պարզապես հատկություն, որը հարկ է հաշվի առնել. երբ հավաքածուն ունի միլիոնավոր տարրեր, `Enum` մոդուլի ֆունկցիաների յուրաքանչյուր կանչ ամբողջական անցում է դրա վրայով։

### Փորձեք ինքներդ

Բացեք iex-ը։ Ստեղծեք `1..20` Range-ը և `Enum.to_list`-ի միջոցով ստուգեք դրա պարունակությունը։ `map`-ով ստացեք բոլոր թվերի քառակուսիները, `filter`-ով դրանցից պահեք միայն 100-ից մեծերը, ապա `count`-ով հաշվեք դրանց քանակը։ `each`-ի միջոցով տպեք որևէ անունների list՝ ամեն մեկը առանձին տողում։ Ստուգեք `Enum.member?(1..10, 10)`-ը և `Enum.member?(1..10, 10.0)`-ը և բացատրեք արդյունքների տարբերությունը։

Վերջում դիտարկեք հետևյալ օրինակը.

```elixir
iex> Enum.sum(Enum.filter(Enum.map(1..10, fn x -> x * x end), fn x -> rem(x, 2) == 0 end))
220
```

Այս արտահայտությունը ճիշտ է աշխատում. նախ հաշվում է քառակուսիները, ապա դրանցից ընտրում զույգերը և վերջում գումարում։ Սակայն այն ընթերցվում է ներսից դուրս, ինչը դժվարացնում է ընկալումը։ Երեք գործողությունը մեկը մյուսի մեջ ներդրված է, և սա բավականին դանդաղեցնում է ծրագրավորողի՝ կոդը կարդալու, վերլուծելու ժամանակը։

### Ի՞նչ է սպասվում առջևում

Հաջորդ հոդվածը կսկսվի հենց այս խնդրի լուծումով։ Կծանոթանանք `|>` pipe օպերատորին, որը գործողությունները դասավորում է ձախից աջ՝ ընթերցման բնական ուղղությամբ։ Այնուհետև կլրացնենք ամենօրյա գործիքակազմը՝ սովորելով `Enum` մոդուլի մի շարք կարևորագույն ֆունկցիաներ ևս։
