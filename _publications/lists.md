## List տվյալների տիպը

Մեր մինչ այժմ սովորած տիպերը մեկական արժեք են պահում՝ մեկ թիվ, մեկ string, մեկ atom։ Բայց իրական ծրագրերում գրեթե միշտ գործ ունենք արժեքների խմբերի հետ՝ օգտատերերի ցուցակ, գնումների ցուցակ, գնահատականների ցուցակ։ Elixir-ում դրա հիմնական գործիքը list-ն է (ցուցակ)։

Եթե գիտեք JavaScript, list-ն առաջին հայացքից ծանոթ կթվա. նույն քառակուսի փակագծերը, ինչ array-ը։ Բայց ներսից այն բոլորովին այլ կերպ է աշխատում, և հենց այդ տարբերությունը հասկանալն է այս հոդվածի գլխավոր նպատակը։

### Ի՞նչ է list-ը

List-ը արժեքների հաջորդականություն է՝ գրված քառակուսի փակագծերում.

```elixir
iex> [1, 2, 3]
[1, 2, 3]
```

Տարրերը կարող են լինել տարբեր տիպերի։ Ահա մի list, որտեղ հավաքվել են մեզ արդեն ծանոթ տիպերը՝ ամբողջ թիվ, string, atom և float.

```elixir
iex> [42, "hello", :ok, 3.14]
[42, "hello", :ok, 3.14]
```

Դատարկ list-ը գրվում է պարզապես `[]`։

### Հիմնական գործողությունները

Երկու list միացվում են `++` օպերատորով, իսկ `--` օպերատորը հեռացնում է տարրեր.

```elixir
iex> [1, 2, 3] ++ [4, 5, 6]
[1, 2, 3, 4, 5, 6]
iex> [1, 2, 3, 2] -- [2]
[1, 3, 2]
```

Ուշադրություն դարձրեք վերջին օրինակին. list-ում երկու հատ 2 կար, բայց `--`-ը հեռացրեց միայն առաջինը։ Ամեն տարր հեռացնում է մեկական համընկնում։

`in` օպերատորը ստուգում է՝ արժեքը կա՞ list-ում, թե՞ ոչ.

```elixir
iex> 2 in [1, 2, 3]
true
iex> 10 in [1, 2, 3]
false
```

Իսկ `length/1` ֆունկցիան վերադարձնում է տարրերի քանակը.

```elixir
iex> length([10, 20, 30])
3
```

### Տարր ավելացնելը սկզբից

Elixir-ն ունի հատուկ գրելաձև list-ի սկզբում տարր ավելացնելու համար՝ `|` նշանը.

```elixir
iex> list = [1, 2, 3]
[1, 2, 3]
iex> [0 | list]
[0, 1, 2, 3]
```

Տարօրինակ չէ՞. ինչո՞ւ հատուկ գրելաձև հենց սկզբից ավելացնելու համար, երբ վերջից ավելացնելու համար այդպիսի բան չկա։ Պատասխանը թաքնված է list-ի ներքին կառուցվածքում։

### Ինչպես է list-ը պահվում հիշողության մեջ

Elixir-ի list-ը կապակցված ցուցակ է (linked list)։ Պատկերացրեք գնացք. ամեն վագոն գիտի ընդամենը երկու բան՝ իր բեռը (տարրի արժեքը) և թե որն է հաջորդ վագոնը։ Ամբողջ գնացքի բեռները զննելու միակ ճանապարհը առաջին վագոնից սկսելն է։

Այստեղից բխում են երկու կարևոր հետևանք։

**Սկզբից ավելացնելը միշտ արագ է։** `[0 | list]` գրելիս Elixir-ը պարզապես ստեղծում է մեկ նոր վագոն, որը ցույց է տալիս արդեն գոյություն ունեցող գնացքին։ Կապ չունի՝ list-ում 3 տարր կա, թե 3 միլիոն. գործողությունը նույնքան ժամանակ է տևում։

**Վերջից ավելացնելը դանդաղ է։** `list ++ [4]` գրելիս Elixir-ը ստիպված է անցնել ամբողջ գնացքով՝ առաջին վագոնից մինչև վերջինը։ Որքան երկար է list-ը, այնքան երկար է տևում դա։

```elixir
iex> list = [1, 2, 3]
[1, 2, 3]
iex> [0 | list]      # արագ
[0, 1, 2, 3]
iex> list ++ [4]     # դանդաղ երկար list-երի դեպքում
[1, 2, 3, 4]
```

Նույն պատճառով `length/1`-ը նույնպես անցնում է ամբողջ շղթայով. տարրերը հաշվելու այլ ճանապարհ չկա։

Իսկ ի՞նչ անել, երբ պետք է list-ի առաջին տարրը։ `arr[0]` գրելաձև այստեղ չկա. դրա փոխարեն կան `hd/1` (head՝ գլուխ) և `tl/1` (tail՝ պոչ) ֆունկցիաները.

```elixir
iex> hd([1, 2, 3])
1
iex> tl([1, 2, 3])
[2, 3]
```

`hd`-ն վերադարձնում է առաջին տարրը, `tl`-ը՝ մնացած ամբողջ list-ը։ Դատարկ list-ը ո՛չ գլուխ ունի, ո՛չ պոչ, ուստի `hd([])` գրելը սխալ է առաջացնում։

Ծրագրավորման մեջ գործողության «արագ» կամ «դանդաղ» լինելը ընդունված է նկարագրել Big O գրառմամբ։ Այն ցույց է տալիս, թե ինչպես է աճում գործողության ժամանակը, երբ աճում է տվյալների քանակը՝ n-ը։ O(1) նշանակում է՝ ժամանակը հաստատուն է և կախված չէ list-ի երկարությունից։ O(n) նշանակում է՝ ժամանակը աճում է list-ի երկարությանը համաչափ. տասն անգամ երկար list՝ մոտավորապես տասն անգամ երկար սպասում։ Մեր սովորածը այս լեզվով կհնչի այսպես. սկզբից տարր ավելացնելը O(1) է, առաջին տարրը վերցնելը (`hd`)՝ նույնպես O(1), իսկ վերջից ավելացնելը, կամայական տարրին հասնելը և `length`-ը՝ O(n)։ Big O-ի մասին ավելի մանրամասն կխոսենք առանձին հոդվածում։

## Charlist-ի գաղտնիքը

Bitstring-ների հոդվածում խոստացանք, որ charlist-ին դեռ կանդրադառնանք։ Հիմա, երբ գիտենք՝ ինչ է list-ը, կարող ենք վերջապես բացել փակագծերը։

Charlist-ը սովորական list է, որի բոլոր տարրերը code point-եր են.

```elixir
iex> ~c"abc"
~c"abc"
```

Եվ ահա այն պահը, որը շփոթեցնում է գրեթե բոլոր սկսնակներին.

```elixir
iex> [104, 105]
~c"hi"
```

Մենք գրեցինք թվերի list, իսկ iex-ը մեզ տեքստ ցույց տվեց։ Ինչո՞ւ։ Երբ list-ը կազմված է միայն տպագրելի ASCII code point-երից (104-ը h տառի code point-ն է, 105-ը՝ i-ի), iex-ը այն ցույց է տալիս `~c` տեսքով։ Տվյալը չի փոխվել. այն մնում է թվերի list, փոխվել է միայն ցուցադրումը։ Համոզվելու համար կարող ենք խնդրել iex-ին ցույց տալ «իրական» տեսքը.

```elixir
iex> inspect([104, 105], charlists: :as_list)
"[104, 105]"
```

Հայերեն տառերի code point-երը ASCII չեն, ուստի դրանց list-ը մնում է սովորական տեսքով.

```elixir
iex> [1377, 1378]
[1377, 1378]
```

Այս պահվածքը ժառանգություն է Erlang-ից, որտեղ տեքստը ավանդաբար պահվում է charlist-երով, մինչդեռ Elixir-ում՝ binary-ներով։ Հին Erlang գրադարանները երբեմն վերադարձնում են charlist, և iex-ի այս ցուցադրումը օգնում է դրանք միանգամից ճանաչել։ Առօրյա կոդում ձեզ գրեթե միշտ string է պետք, ոչ թե charlist։

### Փորձեք ինքներդ

Բերեք նախ մի քանի փոքր վարժություններ անենք iex-ում. ստեղծեք ձեր գնումների ցուցակը, `|`-ով ավելացրեք ամենակարևոր ապրանքը սկզբում, `--`-ով հեռացրեք արդեն գնվածը, `in`-ով ստուգեք՝ մոռացե՞լ եք ինչ-որ բան, `length`-ով հաշվեք մնացորդի քանակը։ Ամեն գործողությունից հետո հարցրեք ինքներդ ձեզ. սա O(1) գործողությո՞ւն էր, թե՞ O(n)։

Ապա երկու դասական խնդիր.

1. **Միջին գնահատական** — [lists_average_grade.exs](https://github.com/vontanne/elixir-hy/blob/master/examples/lists_average_grade.exs)։ Տրված է գնահատականների list. հաշվեք գումարը և միջինը։
2. **List-ի շրջում** — [lists_reverse.exs](https://github.com/vontanne/elixir-hy/blob/master/examples/lists_reverse.exs)։ Տրված է list. ստացեք նույն տարրերը հակառակ հերթականությամբ և ստուգեք՝ list-ը պալինդրո՞մ է (նույնն է՝ երկու կողմից կարդալիս)։

Ինչպես FizzBuzz-ի օրինակում, ֆայլերում կհանդիպեն շարահյուսական տարրեր (`defmodule`, `def`, `IO.puts`), որոնց դեռ չենք անդրադարձել. պարզապես կենտրոնացեք list-երի հետ աշխատող տողերի վրա։
