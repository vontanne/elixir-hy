## Tuple տվյալների տիպը

Արդեն ծանոթ ենք երկու «հավաքածուի»՝ [list-երին](./lists.md) և [map-երին](./maps.md)։ Այսօր կծանոթանանք tuple-ներին։ Առաջին հայացքից այն list-ի կրկնօրինակն է թվում, միայն այլ փակագծերով։ Բայց tuple-ն ուրիշ նպատակով է կիրառվում, և հենց այդ «ինչի՞ համար» հարցին է նվիրված այս հոդվածը։

Tuple-ը գրվում է ձևավոր փակագծերով, և տարրերը, ինչպես list-ում, կարող են լինել տարբեր տիպերի.

```elixir
iex> {}
{}
iex> {1, :two, "three"}
{1, :two, "three"}
```

Գլխավոր տարբերությունը՝ tuple-ը **ֆիքսված չափի** կոնտեյներ է։ List-ը «առաձգական» հավաքածու է, որին անընդհատ տարրեր ենք ավելացնում ու հեռացնում։ Tuple-ը ավելի շուտ մեկ ամբողջության ֆիքսված մասերն է իրար կողքի պահում. կոորդինատներ՝ `{x, y}`, ամսաթիվ՝ `{day, month, year}`, գույն՝ `{red, green, blue}`։ Մասերի քանակը հայտնի է սկզբից և չի փոխվում։

### Տարրերին հասնելը

Տարրը վերցնում ենք `elem/2` ֆունկցիայով՝ նշելով դիրքը։ Ուշադրություն. համարակալումը սկսվում է **0-ից**, ինչպես որ list-երի դեպքում էր.

```elixir
iex> date = {24, 4, 2026}
{24, 4, 2026}
iex> elem(date, 0)
24
iex> elem(date, 2)
2026
```

Չափը տալիս է `tuple_size/1`-ը, իսկ տարրը փոխարինում է `put_elem/3`-ը.

```elixir
iex> tuple_size(date)
3
iex> put_elem(date, 0, 25)
{25, 4, 2026}
iex> date
{24, 4, 2026}
```

Ծանոթ պատկե՞ր է։ `date`-ը մնաց անփոփոխ. `put_elem`-ը վերադարձրեց **նոր** tuple, ճիշտ ինչպես map-երի `%{map | ...}` գրելաձևը նոր map էր ստեղծում։ Elixir-ում այս կանոնը բացառություն չունի՝ տվյալները չեն փոխվում, ստեղծվում են նորերը։

Իսկ եթե դիրքը գոյություն չունի (օրինակ՝ `elem(date, 10)`), կստանաք `ArgumentError`. tuple-ը «լուռ `nil`» չի վերադարձնում։

### Tuple, թե՞ list

Հիշո՞ւմ եք list-երի հոդվածի գլխավոր բացահայտումը. list-ը իրար կապված «վագոնների» շղթա է, դրա համար էլ սկզբից ավելացնելը O(1) է, իսկ կամայական տարրին հասնելը՝ O(n)։ Tuple-ի ներքին կառուցվածքը տարբերվում է. տարրերը հիշողության մեջ պահվում են իրար կողքի, մեկ ամբողջ «շարքով»։

Դրանից բխում է ամեն ինչ.

- Ցանկացած տարրի հասնելը (`elem/2`) O(1) է. Elixir-ը միանգամց գիտի, թե որտեղ է n-րդ տարրը։
- `tuple_size/1`-ը նույնպես O(1) է՝ ինչպես map-երի `map_size/1`-ը (իսկ list-երի `length/1`-ը, հիշում ենք, O(n) էր)։
- Բայց ցանկացած «փոփոխություն» (`put_elem/3` և նման ֆունկցիաներ) O(n) է. քանի որ տարրերը մեկ ամբողջություն են, Elixir-ը ստիպված է ամբողջ tuple-ը պատճենել։

Այստեղից էլ գործնական կանոնը, որը ձևակերպում է հենց պաշտոնական փաստաթուղթը. tuple-ը հարմար է **կարդալու** համար։ Եթե տարրերի քանակը փոխվում է, ձեզ list է պետք։ Եթե տարրերի քանակը նախապես հայտնի է, ու չի փոխվելու կամ մեկ բանի ֆիքսված մասեր են՝ tuple։

### Enum-ը tuple-ների հետ չի աշխատում

`Enum` մոդուլի ֆունկցիաները (մանրամասն կծանոթանանք առաջիկա հոդվածներից մեկում), որոնք list-երի (և map-երի) հետ աշխատում էին, tuple-ների հետ պարզապես հրաժարվում են աշխատել.

```elixir
iex> Enum.sum({255, 165, 0})
** (Protocol.UndefinedError) protocol Enumerable not implemented for type Tuple
```

Սա գիտակցված որոշում է։ Երբ իսկապես պետք է, կա հնարավորություն այն փոխակերպել list-ի՝ `Tuple.to_list/1`-ը կօգնի այդ հարցում.

```elixir
iex> Tuple.to_list({255, 165, 0})
[255, 165, 0]
iex> Enum.sum(Tuple.to_list({255, 165, 0}))
420
```

Փոքրիկ հիշեցում. եթե ձեր tuple-ի թվերը փոքր են, `Tuple.to_list`-ի արդյունքը iex-ում կարող է տարօրինակ տեսք ունենալ։ Դա մեզ արդեն ծանոթ [charlist-ի գաղտնիքն է](./lists.md) list-երի հոդվածից։

`Tuple` մոդուլն ունի նաև տարր ավելացնող ու հեռացնող ֆունկցիաներ (`Tuple.insert_at/3`, `Tuple.delete_at/2`), բայց պաշտոնական փաստաթուղթն ինքն է զգուշացնում. գործնականում դրանք գրեթե չեն օգտագործվում։ Եթե tuple-ին տարրեր եք ավելացնում ու հանում, ամենայն հավանականությամբ ձեզ իրականում list էր պետք։

### Pattern matching-ը tuple-ների հետ

Map-երի հոդվածում ծանոթացանք pattern matching-ի գաղափարին: Tuple-ների հետ դա նույնիսկ ավելի ընթեռնելի և բնական է ստացվում.

```elixir
iex> {day, month, year} = {24, 4, 2026}
{24, 4, 2026}
iex> month
4
```

Փաստացի մեկ տողում ստացանք 3 փոփոխական, և կապեցինք այդ փոփոխականները tuple-ի համապատասխան 
արժեքների հետ։ Բայց ուշադրություն դարձրեք մի կարևոր տարբերության։ Map-ի կաղապարում կարող էինք նշել միայն մեզ հետաքրքրող բանալիները։ Tuple-ի կաղապարն այլ պահանջներ ունի. այն պետք է **ամբողջությամբ** նկարագրի tuple-ը՝ ճիշտ նույն քանակի տարրերով.

```elixir
iex> {day, month} = {24, 4, 2026}
** (MatchError) no match of right hand side value: {24, 4, 2026}
```

Տրամաբանական է. tuple-ը ֆիքսված ձև ունեցող արժեք է, ուրեմն կաղապարն էլ պետք է ունենա ճիշտ նույն ձևը։

### Tuple-ների գլխավոր դերը. {:ok, ...} և {:error, ...}

Հիմա ամենակարևորը։ Elixir-ում Tuple-ների ամենաշատ կիրառությունը ֆունկցիաների վերադարձվող արժեքներն են։ Երբ ֆունկցիան պետք է վերադարձնի ոչ թե մեկ արժեք, այլ «արդյունքը՝ գումարած տեղեկություն արդյունքի մասին», այդ ամենը փաթեթավորվում է tuple-ի մեջ։

Դասական օրինակը ֆայլեր կարդալն է։ `File.read/1` ֆունկցիան հաջողության դեպքում վերադարձնում է `{:ok, պարունակությունը}`, իսկ ձախողման դեպքում՝ `{:error, պատճառը}`.

```elixir
iex> File.read("notes.txt")
{:ok, "hello world"}
iex> File.read("missing.txt")
{:error, :enoent}
```

Առաջին տարրը atom է, որը պատասխանում է «ստացվե՞ց» հարցին, երկրորդը՝ բուն արժեքը կամ ձախողման պատճառը։ `:enoent`-ը հենց այդպիսի պատճառ նկարագրող atom է («error: no entry»՝ ֆայլը գոյություն չունի)։ Այս «պայմանավորվածությանը» հետևում է ամբողջ Elixir աշխարհը. հարյուրավոր ֆունկցիաներ վերադարձնում են հենց `{:ok, ...}` կամ `{:error, ...}`։

Իսկ արժեքները կապում ենք փոփոխականների հետ մեզ արդեն ծանոթ pattern matching-ով.

```elixir
iex> {:ok, contents} = File.read("notes.txt")
{:ok, "hello world"}
iex> contents
"hello world"
```

Կաղապարի առաջին տարրը փոփոխական չէ, այլ կոնկրետ `:ok` atom-ը։ Այսինքն՝ այս տողը միաժամանակ և՛ **ստուգում** է, որ գործողությունը հաջողվել է, և՛ **հանում** է արժեքը։ Եթե ֆայլը չլիներ, աջ կողմում կլիներ `{:error, :enoent}`, `:ok`-ը չէր համընկնի, և կստանայինք `MatchError`։

Իսկ ինչպե՞ս ճիշտ վարվել երկու դեպքում էլ՝ և՛ հաջողության, և՛ ձախողման։ Դրա համար Elixir-ն ունի հրաշալի գործիքներ, որոնց կծանոթանանք առաջիկա հոդվածներում։ Առայժմ բավական է ճանաչել այս զույգերին. այսուհետ դրանք ամեն քայլափոխի եք հանդիպելու։

### Փորձեք ինքներդ

Բացեք iex-ը։ Ստեղծեք ձեր ծննդյան ամսաթիվը պահող `{day, month, year}` tuple-ը։ `elem/2`-ով վերցրեք տարին, `tuple_size/1`-ով ստուգեք չափը, `put_elem/3`-ով «տեղափոխեք» ծնունդը մեկ օր առաջ և համոզվեք, որ սկզբնական tuple-ը մնաց անփոփոխ։ Ապա մեկ տողով, pattern matching-ի օգնությամբ, երեք տարրերը բացեք երեք անունների մեջ։ Փորձեք նաև դիտավորյալ սխալ կաղապար՝ երկու տարրով, և ուշադիր կարդացեք `MatchError`-ի հաղորդագրությունը։

Այնուհետև ստեղծեք `{255, 165, 0}` գույնը (նարնջագույնն է), փորձեք `Enum.sum`-ը ուղիղ tuple-ի վրա, կարդացեք ստացված սխալը, ապա ուղղեք այն `Tuple.to_list/1`-ի օգնությամբ։

### Ի՞նչ է սպասվում առջևում

Այժմ մեր «զինանոցում» երեք հավաքածու կա. list՝ փոփոխվող հավաքածուների, map՝ բանալի-արժեք զույգերի, tuple՝ ֆիքսված ձևերի համար։ Pattern matching-ի կտորներն էլ քայլ առ քայլ շատանում են, և առաջիկայում առանձին հոդվածով կանդրադառնանք նաև դրան։
